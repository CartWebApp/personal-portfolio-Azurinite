EECU Budget Calculator

Clients requested a financial literacy web application which would help young adults transitioning to the workspace plan how to spend their budget.

Users would receive an approximation of the income they make, then fill out expenses per category to find out whether they have a “surplus” or “shortfall” of money.

Problems & Solutions

  • 1. Communication Between Developers

      The Problem:Needing to make sense of previous work other developers did detracted from my next developer’s work time.

      The Solution: I began encouraging developers to comment their code to maintain organization in the CSS file and explain the functionality of JavaScript methods to less knowledgeable developers. This cut down the average time spent reading through code from 20 minutes to 10.

  • 2. Fetch API not properly loading the dropdown

      The Problem: While careers and salaries seemed to be successfully fetched from the Fetch API (appearing in the DOM), any code attempting to read over the <select> dropdown did not seem to be able to access any <option> elements nor their values.

      The Solution: After searching online, a thread on StackOverflow helped me realize that the dropdown must finish loading before proceeding with the rest of the code’s initialization and iteration of the DOM.

  • 3. Improper Progressive Tax Calculations

      The Problem:The Progressive Tax function seemed to not be deducting enough—In fact, a console.log() showed the third bracket was adding money.

      The Solution: The previous code was unnecessarily long, attempting to separately calculate each bracket at once in large sections of code. After researching the Progressive Tax function more thoroughly online and reading through others’ solutions, I found out it would be easier to simply use if statements to collect tax amounts depending on if the user’s salary was above or below bracket thresholds.

  • 4. Scrolling Redundancies

      The Problem:The user would have to constantly scroll their screen between the next/back buttons and the input fields.

      The Solution: Decreased the height of elements to allow the next/back bars and input fields to fit as compactly as they could in the viewport. Added an event listener for page transition to automatically scroll the page to put the next/back buttons in view, allowing the user to easily click through several pages without having to scroll.
